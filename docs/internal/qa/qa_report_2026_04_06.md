## New Findings

### MEG-050 -- Combined pytest invocation fails: duplicate test_license.py basenames

**Severity:** YELLOW
**Status:** NEW
**File:** ron_skills/loreconvo/tests/test_license.py AND ron_skills/loredocs/tests/test_license.py

When running the three suites together in a single `pytest` command:

```
pytest ron_skills/loreconvo/tests/ ron_skills/loredocs/tests/ tests/
```

pytest fails at collection with:

```
ERROR ron_skills/loredocs/tests/test_license.py
HINT: remove __pycache__ / .pyc files and/or use a unique basename for your test file
modules
```

The root cause is that both product test directories contain a file named
`test_license.py`. Without `__init__.py` files (or with conflicting module paths),
pytest cannot distinguish them in a combined run and raises an import collision.

**Impact:** Any CI script or developer command that attempts to run the full test suite
in one invocation will fail at collection. Each suite must be run independently as a
workaround.

**Options for Ron:**
1. (Preferred) Add `__init__.py` to each test directory so pytest treats them as
   packages with distinct import paths. This is the standard fix for this class of
   naming collision.
2. Rename one file: `test_license_loreconvo.py` / `test_license_loredocs.py`.
   Simpler but less conventional.

**Workaround:** Run each suite separately:
```
pytest ron_skills/loreconvo/tests/ -q
pytest ron_skills/loredocs/tests/ -q
pytest tests/ -q
```

## Open Carry-Forward Items

### GINA-003 -- license.py dev_mode divergence (open, Ron's work item)

**Severity:** MEDIUM (from Gina's report)
**Status:** OPEN -- not yet fixed
**File:** ron_skills/loredocs/loredocs/license.py line 169 and 201

LoreDocs `get_license_status()` is missing the `env_value and` guard that LoreConvo
has:

```
# LoreConvo (correct):
if dev_mode and env_value and not env_value.startswith(_KEY_PREFIX):

# LoreDocs (missing guard):
if dev_mode and not env_value.startswith(_KEY_PREFIX):
```

When `LAB_DEV_MODE=1` and `LOREDOCS_PRO` is unset (empty string), LoreDocs
`"".startswith(_KEY_PREFIX)` evaluates to `False`, so the condition is `True` and
LoreDocs grants Pro bypass. LoreConvo does NOT grant bypass in this case (the
`env_value and` guard short-circuits). This is a behavioral inconsistency: dev mode
with no key set grants Pro in LoreDocs but not LoreConvo.

**Fix:** Add `env_value and` to LoreDocs license.py line 169 and 201 (two places).
1-line fix per occurrence.

**Note:** Gina also flagged a BROCK-REVIEW on this item to confirm intended behavior.
Ron should fix to match LoreConvo's stricter behavior.
